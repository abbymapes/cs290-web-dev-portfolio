Data

To implement BlueBook, I will use the following data:

Class Data
In order to drive my web app, I will use data from the Duke Curriculum API, which provides JSON streams of data that contain various course details. Since this API is very slow to fetch information for multiple classes at a time, I’ve written a code to preload the necessary data I need for my web app. I’ve stored the following fields from the API for each course: catalogNumber, courseCodes, courseId, description, offerNumber, subjectCode, subjectName, title, and type. Along with the fields from the Duke Curriculum API, I’ve added some fields to drive my web app– including a count of “interesting” ratings and a sum of “interesting” ratings that I will use to calculate an average “interesting” rating for each course, based on user inputted ratings. Additionally, I added a “difficulty” sum and “difficulty” count to do the same for a difficulty rating. Here is the JSON data structure I will use for each course.

User Data
In order to display user profiles, I need to store the following information for each user: uid, displayName, email, picture, majors, minors, certificates, subjects. This information will all be inputted by the user when creating their profile, or by liking subjects when interacting with Bluebook. Here is the JSON data structure I will use for each user.

Reaction Data
In order to store user reactions to different class, I need to store the following information for each time a user reacts to a class (by either liking, disliking, or adding the class to their wishlist): type, userId, courseId, reactionId, date, courseTitle, catalogNumber, subjectCode. This information will allow me to display a reaction component (either for each user or for the people they are following) without having to fetch information from other collections in Firestore. Here is the JSON data structure I will use for each reaction.

Follows Data
For each user, I will store following information in order to retrieve a list of their friends and the class reactions that will fill their feed. For each following relationship, I will store the followerId and the followingId, which correspond to the user Id of person A following the userId of person B. This data will be entered by users when they follow or unfollow each other. Here is the JSON data structure I will use for each follows relationship.

Comment Data
For each comment made on a reaction post or a course page, I will store the following information for each comment: reactionId, courseId, userId, commentText, date. This data will be entered by users when they post comments on other people’s reaction posts or on a course page. Here is the JSON data structure I will use for each comment.

Subjects Data
For each subject in this data structure, which I’ve created based on the subjects listed in Duke’s registration website, I stored a document containing a subjectCode and displayName property in Firestore. This will allow us to fetch display names for subjects based on their subject codes from a user’s list of subjects or a course’s subjectCode data. Here is the JSON data structure I will use for subjects data.

Progress

Based on my plan for Weeks 1 plan, I planned to complete the following features:

  • Set up Firestore database
    • I loaded the Firestore database with course information from Duke API– this took longer than expected due to the slowness of the API itself to fetch each class.
  • Set up Home, Log In & Sign Up Screens
    • I haven’t started the homepage yet. But, I may not add the homepage and instead send users straight to the explore page (which is available to both guests and users). When actually implementing my app within my portfolio, I realized that a homepage wouldn’t really be necessary. Additionally, I haven’t begun the login page or the sign up page. Again, once starting I realized it would be easier to work on the rest of the functionality itself, since it will be easier during development to have full access to the site without having to log in or sign up first. I also wanted to wait until we had more experience with OAuth in class to see the best way to do this, rather than spending extra time figuring it out before we talked about it.
  • Create Navigation Bar
    • I’ve created a custom NavBar component that has the functionally set up to limit users based on whether or not they are logged in (which will be a global variable). For dev purposes, I’ve disabled this until I create the log in and sign up pages.

Based on my week 2 plan, I planned to create these features:

  • Set up pages, but don’t need to implement reactivity / connecting to database yet
